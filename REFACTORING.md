# Refactoring Opportunities — Shop Floor Resource Allocation

> Analysis date: 2026-05-26

---

## Critical — Bugs & Security

| # | File | Issue |
|---|---|---|
| C1 | `backend/src/services/workOrderService.ts` | Invalid status transition throws plain `Error` → hits 500 handler, should return 400 |
| C2 | `backend/src/socket/index.ts` | No JWT authentication on socket connection — anyone who knows the server URL can join the `shop-floor` room and receive all real-time events without being logged in |
| C3 | `backend/src/services/authService.ts` | `process.env.JWT_SECRET!` and `JWT_REFRESH_SECRET!` use non-null assertions validated at call time, not startup. An unset env var silently causes runtime 500s instead of a clear startup failure. Fix: validate all required env vars in `index.ts` before binding the server |
| C4 | `frontend/src/services/api.ts` | On 401 expiry, the interceptor redirects to `/login` but does **not** clear Zustand stores. Sensitive allocation/operator/work-order data persists in memory after the session ends |

## High — Type Safety & Correctness

| # | File | Issue |
|---|---|---|
| H1 | `frontend/src/types/index.ts` | `Allocation.createdBy: string` should be `createdById: string` — Prisma scalar field is `createdById`; using the wrong name risks silent mismatches when reading from API responses |
| H2 | `backend/src/services/allocationService.ts` | Material stock updates inside `create` and `update` run with sequential `await` inside a transaction. Each iteration targets a different row, so they can run in parallel with `Promise.all`. Same issue in `checkMaterialStock` |
| H3 | `backend/src/services/operatorService.ts`, `machineService.ts`, `workOrderService.ts` | Local `type OperatorStatus = ...` / `MachineStatus` / `WorkOrderStatus` re-declare types already generated by Prisma. Import from `@prisma/client` to eliminate drift risk |
| H4 | All fetch hooks (`useWorkOrders`, `useAllocations`, `useOperators`, `useMachines`, `useMaterials`) | No `.catch()` / error handling at all. API failures are silent — stores stay empty with no user feedback |
| H5 | `frontend/src/components/AllocationBoard/AllocationCard.tsx` | Time editing sends PATCH without validating `endTime > startTime`. DateTime input binding uses `.slice(0, 16)` string manipulation which is fragile if the API returns a different ISO format |
| H6 | `frontend/src/components/AllocationBoard/index.tsx` — `handleNewAssignment` | Does not verify the operator/machine is still `AVAILABLE` at drag-drop time — races with concurrent sessions |
| H7 | `backend/src/services/authService.ts` | JWT `expiresIn` options passed as `as any` casts, bypassing `SignOptions` type safety |

## Medium — Duplication & Architecture

| # | File | Issue |
|---|---|---|
| M1 | `backend/src/services/allocationService.ts` — `checkOperatorOverlap` / `checkMachineOverlap` | Functionally identical ~10-line functions — differ only by the field name. Parameterise into a single `checkResourceOverlap(tx, field, id, start, end, excludeId?)` |
| M2 | `backend/src/controllers/allocationController.ts` | Socket `resource:statusChanged` emission block duplicated verbatim in `create` and `update`. Extract to `emitResourceStatusEvents(allocation)` helper |
| M3 | All five fetch hooks | Identical `setLoading → api.get → setter → finally(setLoading(false))` pattern. Extract a generic `useFetchOnMount<T>(url, setter, setLoading)` hook |
| M4 | `frontend/src/components/AllocationBoard/index.tsx` | Optimistic update/rollback pattern (`prev` snapshot → apply → catch → restore) is repeated three times across `handleDragEnd`, `handleDelete`, and `handleTimeEdit`. Extract into a reusable utility |
| M5 | `frontend/src/components/AllocationBoard/index.tsx` | 180-line component mixes drag orchestration, new assignment, deletion, and time editing. Split into a `useDragAllocation` hook and decomposed event handlers |
| M6 | `AllocationBoard/ResourcePanel.tsx` vs `Resources/index.tsx` | Both are conceptually "resource panels" — name collision causes confusion. Rename the board one to `AssignmentPanel` or `ResourceAssignmentPanel` |
| M7 | `backend/src/errors/ConflictError.ts` | `conflicts: unknown[]` is too broad. Should be a typed union of the three concrete conflict shapes: `OperatorConflict`, `MachineConflict`, `InsufficientStockConflict` |
| M8 | `backend/src/services/allocationService.ts` | `TxClient` type is defined inline. It belongs in `backend/src/lib/prisma.ts` alongside the Prisma instance so other services can reuse it |
| M9 | `backend/src/services/operatorService.ts`, `machineService.ts` | `setStatus` signatures use `Extract<OperatorStatus, 'ABSENT'>` — this is equivalent to just `'ABSENT'`. Remove the unnecessary generic |
| M10 | `backend/src/services/workOrderService.ts` | `update` builds the update object as `Record<string, unknown>`, losing type safety. Use `Prisma.WorkOrderUpdateInput` directly |

## Low — Code Quality & Conventions

| # | Issue |
|---|---|
| L1 | All fetch hooks use no `AbortController` — navigating away mid-fetch causes `setState on unmounted component` React warnings |
| L2 | Axios instance (`frontend/src/services/api.ts`) has no `timeout` configured — requests can hang indefinitely |
| L3 | `frontend/src/hooks/useSocket.ts` doesn't handle `connect_error` or reconnect events — socket failures are completely silent |
| L4 | No `ErrorBoundary` in `frontend/src/App.tsx` — an unhandled React render error crashes the entire app |
| L5 | `backend/src/controllers/authController.ts` sets `sameSite: 'strict'` on the refresh cookie. This can block the cookie from being sent on top-level navigations. `'lax'` is the conventional choice for this auth pattern |
| L6 | `frontend/src/store/resourceStore.ts` has no computed selectors. `operators.filter(o => o.status === 'AVAILABLE')` is inlined across multiple components, re-running the filter on every render |
| L7 | `frontend/src/store/allocationStore.ts` — `addAllocation` silently performs either an insert or an update depending on whether the item already exists. The dual behaviour is undocumented and inconsistent with how `addWorkOrder` works |
| L8 | Prisma schema is missing an index on `Allocation.workOrderId`. The board filters allocations by `workOrderId` on every render; without an index this is a full table scan |

## Recommended Fix Order

1. **C2** — Socket auth (unauthenticated access to live data)
2. **C3** — Env var startup validation (silent production crashes)
3. **H4** — Fetch hook error handling (every data-load silently discards failures)
4. **C4** — Clear stores on session expiry
5. **C1** — Invalid transition status code (500 → 400)
6. **H1** — `createdBy` / `createdById` type mismatch
7. **H2** — `Promise.all` for parallel material updates
8. **M1** — Merge duplicate overlap-check functions
9. **M2** — Extract `emitResourceStatusEvents` helper
10. **M3** — Generic `useFetchOnMount` hook
